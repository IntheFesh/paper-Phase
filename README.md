# Phase-Centric Vision-Language-Action Policy

**A phase-centric policy for long-horizon robotic manipulation.**
Built on the [LeRobot](https://huggingface.co/docs/lerobot/index)
plugin architecture, targeting the
[LIBERO](https://libero-project.github.io/) benchmark, this repo
ships a **four-layer multimodal, hierarchical, closed-loop
generative policy** with a family of phase-centric innovations.

The through-line is to promote phase from a passive feature to a
**first-class control variable** — driving representation
(identifiability), loss (boundary weighting), and online decision
(adaptive replanning).

**Method contribution**: a four-layer closed-loop policy plus six
phase-centric innovations (I1–I6), jointly optimising
identifiability, boundary weighting, density curriculum, and
budget-quantile replanning.

**Core idea**: $\beta_t$ is a **shared signal** threaded through
training and inference — upstream InfoNCE produces an identifiable
phase $z$; the Bhattacharyya distance then converts $z$ into a
continuous boundary posterior $\beta_t \in [0,1]$; downstream
training losses (PACE-A/B/C) and online replanning (PCAR) share
the same $\beta_t$.

### 1.3 LIBERO-Long vs LIBERO-Spatial: the phase-structure falsification

On LIBERO-Spatial, PACE-C (phase-density curriculum) gives
$\Delta = -1.9$ pp (80.2% → 78.3%) — this is **expected** rather
than a regression. Spatial tasks are short-horizon with no obvious
phase boundaries, so the stage-wise $n_b$ filter discards valid
samples without an offsetting benefit. The same config is neutral
or slightly positive on Long tasks. This contrast **shows the
gains come from phase structure, not a generic regulariser**: if
PACE-C also helped on Spatial, the "curriculum = general-purpose
denoising" alternative explanation could not be ruled out. The
full Spatial data is at
[`docs/innovations/round-8-ablation-table-spatial.csv`](docs/innovations/round-8-ablation-table-spatial.csv).

## 2. Architecture at a glance

```
  Observation  x_t = {V_t, S_t, L, H_t}
        │
        ▼
  ┌──── Layer 1 ─ DualBackboneVisionTokenizer ───────────────────────┐
  │ SigLIP2 + DINOv2  (LoRA r=16, α=32, DoRA)                        │
  │ Prismatic concat → 1×1 Conv fusion                               │
  │ Frozen T5-Base + FiLM (γ, β)  →  Perceiver readout (8 queries)   │
  │ Uncertainty gate σ(MLP[readout, state])                          │
  └──────────────────────────────┬──────────────────────────────────┘
                                 │ fused_obs (B, 256)
                                 ▼
  ┌──── Layer 2 ─ HierarchicalPlanner ──────────────────────────────┐
  │ FSQ (levels=[8,6,5], K=240)  →  phase_id,  phase_embed (B, 32)   │
  │ MLP(256→32)                   →  skill_latent (B, 32)            │
  └──────────────────────────────┬──────────────────────────────────┘
                                 │
                                 ▼
  ┌──── Layer 3 ─ ShortcutFlowActionHead  (1-NFE inference) ─────────┐
  │ SmallDiT : 4 × AdaLN-Zero,  hidden=256,  heads=8                 │
  │ Shortcut self-consistency (Frans et al., arXiv 2410.12557)       │
  │ Output :  A_t ∈ R^{16 × 16}                                      │
  └──────────────────────────────┬──────────────────────────────────┘
                                 │
                                 ▼
  ┌──── Layer 4 ─ IQLChunkVerifier ─────────────────────────────────┐
  │ V_ψ expectile τ=0.8,  Q_θ TD(0) γ=0.99                           │
  │ confidence = σ(β_c · (Q − V)),   should_replan = (conf < 0.5)    │
  └──────────────────────────────┬──────────────────────────────────┘
                                 │
                                 ▼
          BID sampler (N=5)  →  ACT temporal ensembling  →  a_t
```

**Tensor shapes, short form**: inputs $(B, V{=}2, 3, 224, 224)$
dual-view RGB and $(B, 8)$ proprioception are mapped by L1 to
`fused_obs` $(B, 256)$; L2 produces `phase_id` $(B,)$,
`phase_embed` $(B, 32)$, `skill_latent` $(B, 32)$; L3 outputs an
action chunk $(B, T_a{=}16, D_a{=}16)$; L4 returns a
`chunk_confidence` $(B,)$ per chunk; at rollout the BID sampler
produces $(N{=}5, T_a, D_a)$ and the ensembler collapses it to
$(D_a,)$. Full tensor shape table at
[`docs/ARCHITECTURE.md`](docs/ARCHITECTURE.md) §1.2.

## 3. Core innovations

Phase-Centric VLA sits on top of PhaseQFlow++ and adds six
phase-centric innovations. Each is controlled by an explicit
feature gate in `PhaseQFlowConfig`, off by default, for clean
ablation.

| # | Name | Problem addressed | Core formula | Related work |
| :-- | :-- | :-- | :-- | :-- |
| I1 | **Chunk-level InfoNCE** | phase codebook collapse, no cross-seed comparability | $-\log\frac{\exp(s_{ii}/\tau)}{\sum_j \exp(s_{ij}/\tau)}$ | iVAE (arXiv [1907.04809](https://arxiv.org/abs/1907.04809)); CPC (arXiv [1807.03748](https://arxiv.org/abs/1807.03748)) |
| I2 | **Bhattacharyya posterior $\beta_t$** | needs a continuous comparable boundary signal | $1 - \sum_k \sqrt{\hat p_t(k)\,\hat p_{t-1}(k)}$ | Hellinger / TV inequalities |
| I3 | **PACE-A** (weighted FM loss) | boundary-step error gets diluted by uniform MSE | $\mathcal{L} = \mathbb{E}[(1+\lambda\beta)\|v-v^*\|^2] - \eta H(\beta)$ | Focal Loss (arXiv [1708.02002](https://arxiv.org/abs/1708.02002)) |
| I4 | **PACE-B** (phase-gated MoE) | single velocity network carrying all phases | $g_t = \sigma(\kappa\beta-\mu)p_t + (1-\cdot)g_{t-1}$ | Switch Transformer (arXiv [2101.03961](https://arxiv.org/abs/2101.03961)) |
| I5 | **PACE-C** (phase-density curriculum) | boundary-dense episodes disturb early-training gradients | stage-wise $n_b$ filter | Curriculum Learning (Bengio et al., ICML 2009) |
| I6 | **PCAR** (budget-adaptive replanning) | a fixed trigger threshold is unstable across tasks | $\tau^{\text{cp}} = \hat Q_n(1-\epsilon)$ (DKW guarantee) | Adaptive Computation Time (arXiv [1603.08983](https://arxiv.org/abs/1603.08983)) |

### 3.1 Theoretical pillars (how the four pieces connect)

- **§2.1 Identifiability** — Chunk-level InfoNCE aligns the phase
  code $z$ across seeds/episodes, prevents VQ collapse, and
  anchors every downstream posterior.
- **§2.2 Posterior $\beta_t$** — the Bhattacharyya distance
  quantifies the gap between adjacent $\hat p_t$ values as a
  continuous, normalised boundary signal that can be updated
  step-by-step during rollout.
- **§2.3 PACE-A weighted FM** — weights the Flow Matching loss by
  $w=1+\lambda\beta$ so boundary steps get a gradient budget
  proportional to their difficulty.
- **§2.4 PCAR budget trigger** — quantises the trigger threshold
  as $\tau^{\text{cp}}=\hat Q_n(1-\epsilon)$, with the DKW
  inequality guaranteeing the cross-task replan rate converges to
  $\epsilon \pm O(1/\sqrt{n})$.

**The shared $\beta$** is what separates this family from ad-hoc
heuristics like PhaseNet or MoE routers: training (PACE-A/B/C) and
inference (PCAR) pull from the same signal. Full derivation and
the connection diagram are in
[`docs/ARCHITECTURE.md`](docs/ARCHITECTURE.md) §2.5.

Underlying components: Shortcut Models
(arXiv [2410.12557](https://arxiv.org/abs/2410.12557)),
DiT AdaLN-Zero (arXiv [2212.09748](https://arxiv.org/abs/2212.09748)),
FSQ (arXiv [2309.15505](https://arxiv.org/abs/2309.15505)),
IQL (arXiv [2110.06169](https://arxiv.org/abs/2110.06169)),
DoRA (arXiv [2402.09353](https://arxiv.org/abs/2402.09353)),
BID (arXiv [2408.17355](https://arxiv.org/abs/2408.17355)),
ACT (arXiv [2304.13705](https://arxiv.org/abs/2304.13705)),
A2C2 (arXiv [2509.23224](https://arxiv.org/abs/2509.23224)).

## 4. Installation and environment

### 4.1 Basic install

```bash
conda env create -f environment.yml
conda activate lerobot_env
pip install -r requirements.txt
pip install -e ./lerobot_policy_phaseqflow
```

Works with `venv` / `uv`. Requires Python $\geq 3.10$ and PyTorch
2.x. On an RTX 5070 we rely on PyTorch's built-in SDPA (**no
flash-attn dependency**).

### 4.2 Dependency notes

| Dependency | Purpose |
| :-- | :-- |
| `timm` | DINOv2/SigLIP vision backbone |
| `peft` | LoRA/DoRA adapters |
| `transformers` | T5-Base language encoder + schedule helpers |
| `vector-quantize-pytorch` | FSQ discrete phase codebook |
| `scipy` | paired-t statistics and DKW quantile estimation |
| `bitsandbytes` *(optional)* | PagedAdamW8bit; falls back to `torch.optim.AdamW` |

```bash
TOTAL_STEPS=3 SEEDS="42 123" DEVICE=cpu SKIP_EVAL=1 \
    bash scripts/training/run_ablation.sh
```

The goal is to confirm the 36-run plumbing is green;
`eval_results.json` will keep `placeholder=true`.

```bash
TOTAL_STEPS=20000 SEEDS="42 123 2024" DEVICE=cuda \
    bash scripts/training/run_ablation.sh
```

Expected time: $36\ \text{runs} \times \sim 8\ \text{h} \approx$
**12 GPU-days** on an RTX 5070.
